## Глоссарий

## Сущности
### 1. Cart (Корзина) 
Внутренняя сущность нашего микросервиса. Формируется на основе входных данных от Core.

Свойства:
- ID. Идентификтор корзины в Core
- weight. Вес корзины в граммах.
- size. Габариты корзины в см3.
- amount. Сумма корзины в рублях. **Вопрос 1**: планируется ли в обозримом будущем поддержка нескольких валют? 
Проектирование системы с учетом нескольких валют может занять весомое количество времени, поэтому важно решить это сразу.

Важно подметить, что интерфейс корзины обозначен в функциональных требованиях и мы хотим его оставить внутри нашей подсистемы 
чтобы иметь возможность сохранить Ubiquitous Language между отделами разработки Core, DeliveryService и системными аналитиками.

### 2. Address (Адрес)
Формат данных адреса не обозначен в ТЗ, поэтому предположим, что там будет набор координат X, Y - минимальный 
набор свойств для точного определения местоположения. **Вопрос 2**: отвечает ли требованиям интерфейс?

Свойства:
- x. Положение по X в float.
- y. Положение по Y в float.

### 3. DeliveryProviderType (Тип провайдера)
Enum, будет определять все доступные в системе провайдеры. Из минусов такого решения - нам придется заново 
компилировать и деплоить наше приложение если мы захотим добавить новый провайдер, но это имеет смысл поскольку 
при добавлении провайдера по-любому придется писать код - динамичных данных в БД недостаточно.

### 4. DeliveryOption (Вариант доставки)
Вариант доставки.

Свойства:
- ID
- providerType. Тип провайдера который будет ответственнен за доставку
- price. Стоимость варианта доставки. Также вопрос с валютами
- deliveryAt. Timestamp со временем ориентировочного выполнения доставки. **Вопрос 3**. В ТЗ ничего не указано 
про алгоритм подсчета времени и не указано будет ли это являться зоной ответственности нашего МС или нет. 
Предположим что это делает наш DeliveryService и придумаем простейший mock-алгоритм. 

### 5. DeliveryProvider (Провайдер доставки)
Одна из центральных сущностей в системе - основная задача которой предоставить интерфейс для формирования DeliveryOptions,
а также будет являться Anti-Corruption Layer для работы с внешними API.

Выделим интерфейс DeliveryProvider от которого будут реализованы классы инкапсулирующие логику работу с конкретным API.

Реализации: MockCdekProvider, MockBoxberryProvider

Методы:
- GetDeliveryOptions(cart: Cart, address: Address): DeliveryOptions[]

Обязанность impl провайдеров будет кастить Cart, Address, DeliveryOptions к данным с которыми работает API.

### 6. DeliveryService (Сервис для работы с доставкой)
Это будет сервис, который будет опрашивать DeliveryProvider и отдавать DeliveryOptions.

## Use cases
Всего один сценарий использования: получить возможные варианты доставки для корзины и адреса. 

## Архитектура
Слои:
- Application. Описывает use-cases. Слой для оркестрации сущностей, работы с DAL и кешем. 
Обычно в этом слое не должно быть Framework, но мы упростим infra слой. Провайдеры тоже будут жить тут.
- Domain. Тут определим основные доменные сущности и интерфейсы. Слой не будет зависеть не от других слоев, не от библиотек, не от фреймворка.
- Infrastructure. Слой инфраструктуры. Будет содержать реализации Data-Access Layer, 
предоставлять абстракцию над транспортом для REST.
- Presentation. Слой где будут лежать контроллеры, DTO и логика работы с ними

## Надежность
Отвечая на вопрос об отказоустойчивости. При работе с внешними API мы должны применить следующие паттерны в коде:
1. Timeout. Любой запрос к API или RPC должен иметь таймаут
2. Ретраи. Если от API пришла сетевая ошибка (ECONNREFUSED, connection reset, timeout), 503, 502, 429 (с retry-after)
3. Circuit Braker. Его применяем для работы с API которому нам нужно дать восстановиться и не флудить запросами.
Это то что мы хотим сделать если API одного из провайдеров «лежит» или отвечает слишком долго. Делаем алерт или лог обязательно про инцидент
4. Graceful Degradation. Если один или несколько из провайдеров сработали с ошибкой, то мы все равно должны отдать пользователю ответ с 
успешными результатами работы других провайдеров, а не просто 500.
5. Graceful Shutdown. Приложение должно уметь высвобождать ресурсы даже в случае непредвиденного завершения.

Следующим важным звеном в обеспечении отказоустойчивости станет инфраструктура. Наше приложение должно быть stateless, 
желательно крутиться в кубере чтобы иметь автоподнятие и авто-масштабирование под нагрузку. Если используем брокер и кеш, 
то они должен быть задеплоены в кластерном режиме чтобы диск не стал точкой отказа. База данных должна иметь реплики и дампы, 
шардирование по мере надобности (много write).
